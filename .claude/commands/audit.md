# Code Review

コードベースを分析し、本当にベストな設計・実装を提案する。

## 前提

このレビューを実行するのはClaude Codeだ。人間のエンジニアではない。つまり:

- 時間は実質無限にある
- コードベース全体を一貫して読める
- 疲れない、見落とさない
- 提案した改善を自分で実装できる

**「工数がかかるから見送り」という判断は不要。** 純粋に「最も保守しやすい設計とは何か」を追求する。

## 目的

**現状のコードを、将来の自分（または他のエンジニア）が最も理解しやすい形にする。**

「動いているから触らない」は理由にならない。「理解しやすいか」「変更しやすいか」が基準。

## 分析観点

### 1. デッドコード

使われていないコードは削除する。「いつか使うかも」は幻想。

- どこからもimportされていないexport
- 受け取っているが使われていないprops
- 到達しない条件分岐
- コメントアウトされたコード

### 2. 責務の分離

1つのモジュールは1つのことをやる。

- 複数の役割を持つコンポーネント → 分割
- 巨大なカスタムフック → 単機能フックに分解
- 「なんでも屋」のユーティリティ → 目的別に整理

### 3. 適切な抽象化

抽象化は「3回同じコードを書いたら」が目安。それ未満は具体的なままでいい。

- 1-2箇所の類似コード → 放置（過度な抽象化は害）
- 3箇所以上の重複 → 共通化を検討
- 使われていない汎用性 → 削除して具体化

### 4. データの流れ

データは一方向に、最短経路で流れるべき。

- Prop drilling → Context または コンポーネント構造の見直し
- 同じデータの複数管理 → Single source of truth
- 不要な中間状態 → 削除

### 5. 計算の配置

計算は適切な場所で行う。

- フロントで重い集計 → バックエンドAPI / Server Componentへ
- 毎レンダリング再計算 → メモ化
- サーバーでやる必要のない処理 → フロントへ

### 6. 型の活用

TypeScriptを使うなら、型で守れるものは型で守る。

- `any` の排除
- Union型で状態を表現
- 型ガードの適切な使用

### 7. 命名と構造

コードは読み物。読みやすさが最優先。

- 意図が伝わらない名前 → リネーム
- 深いネスト → 早期リターンでフラット化
- 長い関数 → 意味のある単位で分割

## レビュー手順

### Phase 1: 全体構造の把握
- ディレクトリ構造とファイル一覧
- 主要な依存関係
- エントリーポイントからのデータフロー

### Phase 2: 静的分析
- 全export/importの追跡
- 未使用コードの検出
- 循環依存の検出

### Phase 3: コンポーネント分析
- 各コンポーネントのprops使用状況
- 責務の数と行数
- 再利用パターン

### Phase 4: フック分析
- 各フックの責務
- 依存関係の正確性
- 状態管理パターンの一貫性

### Phase 5: パフォーマンス分析
- 重い計算の特定
- 不要な再レンダリング
- データ取得パターン

## アウトプット

```markdown
# コードレビュー結果

## 現状の評価

[コードベースの現状を客観的に評価。良い点も明記]

## 理想の設計

[このコードベースが本来あるべき姿を描く]

## ギャップと改善計画

### 1. [改善項目]

**現状**:
// 問題のあるコードを引用

**問題**:
なぜこれが保守性を下げるか

**改善後**:
// あるべき姿のコード

**手順**:
1. 具体的な作業ステップ
2. ...

---

[以下、改善項目ごとに繰り返し]

## 実行順序

依存関係を考慮した実行順序:
1. [最初にやるべきこと]（他の改善の前提になる）
2. [次にやること]
3. ...

## 改善しないこと

以下は意図的に現状維持:
- [項目]: [理由 - 「工数」以外の本質的な理由]
```

## 判断基準

### 改善する
- 理解しにくい → 理解しやすくする
- 変更しにくい → 変更しやすくする
- 重複している → 共通化する（3箇所以上）
- 不要なもの → 削除する

### 改善しない
- 動作に問題がない命名の「好み」の違い
- 過度な抽象化につながる共通化（1-2箇所の類似）
- フレームワークの制約による設計
- 外部ライブラリのインターフェース

## 開始

対象スコープを指定してください:
- 指定なし: src/ 全体
- パス指定: 特定のディレクトリ/ファイル
- `hooks` / `components` / `features`: カテゴリ指定
